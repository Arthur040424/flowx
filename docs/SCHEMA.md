# FlowX - Database Schema

## Overview

FlowX uses PostgreSQL as its primary database. The schema is split into two concerns - **blueprints** (what a pipeline should do) and **execution** history (what actually happened when it ran). These are kept separate so historical data can be managed independently without affecting pipeline definitions.

---

## Tables

### Pipelines

Stores the pipeline definition. One row per pipeline.

| Column        | Type      | Description                              |
| ------------- | --------- | ---------------------------------------- |
| id            | UUID      | Unique identifier, generated by database |
| name          | TEXT      | Human-readable name for the pipeline     |
| cron_schedule | TEXT      | Optional cron expression for scheduling  |
| status        | TEXT      | active or inactive                       |
| created_at    | TIMESTAMP | Set automatically on creation            |
| updated_at    | TIMESTAMP | Updated automatically after every change |

---

### pipeline_Steps

Stores the individual steps belonging to a pipeline. One row per step.

| Column      | Type      | Description                                      |
| ----------- | --------- | ------------------------------------------------ |
| id          | UUID      | Unique identifier, generated by database         |
| pipeline_id | UUID      | Foreign key referencing pipelines.id             |
| name        | TEXT      | Human-readable name for this step                |
| type        | TEXT      | Step type — http, sql, or transform              |
| step_order  | INTEGER   | Execution order within the pipeline (1, 2, 3...) |
| config      | JSONB     | Step-specific configuration stored as JSON       |
| created_at  | TIMESTAMP | Set automatically on creation                    |

---

### pipeline_runs

Records each execution of a pipeline. One row per run.

| Column       | Type      | Description                                     |
| ------------ | --------- | ----------------------------------------------- |
| id           | UUID      | Unique identifier, generated by database        |
| pipeline_id  | UUID      | Foreign key referencing pipelines.id            |
| trigger      | TEXT      | How the run was started — manual or scheduled   |
| status       | TEXT      | pending, running, completed, or failed          |
| started_at   | TIMESTAMP | When execution began                            |
| completed_at | TIMESTAMP | When execution finished — null if still running |

---

### step_results

Records what happened to each step during a specific run

| Column          | Type      | Description                                     |
| --------------- | --------- | ----------------------------------------------- |
| id              | UUID      | Unique identifier, generated by database        |
| pipeline_run_id | UUID      | Foreign key referencing pipeline_runs.id        |
| step_id         | UUID      | Foreign key referencing pipeline_steps.id       |
| status          | TEXT      | pending, running, completed, or failed          |
| error_message   | TEXT      | Populated when status is failed                 |
| started_at      | TIMESTAMP | When this step began executing                  |
| completed_at    | TIMESTAMP | When this step finished — null if still running |

---

## Relationships

pipelines → pipeline_steps (one pipeline has many steps)
pipelines → pipeline_runs (one pipeline has many runs)
pipeline_runs → step_results (one run has many step results)
pipeline_steps → step_results (one step has many results across runs)

## Design Decisions

- Blueprint tables (pipelines, pipeline_steps) are separated from execution tables (pipeline_runs, step_results) so run history can be archived or deleted without affecting pipeline definitions.

- step config is stored as JSONB rather than individual columns because different step types require different configuration fields. JSONB allows flexible, queryable JSON storage in PostgreSQL.

- All IDs use UUID rather than auto-incrementing integers to avoid exposing sequential IDs in URLs, which can leak information about the volume of data in the system.
